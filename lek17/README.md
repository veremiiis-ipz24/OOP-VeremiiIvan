# Лекція №17

## Тема: Породжувальні патерни --- Factory Method і Singleton

------------------------------------------------------------------------

## 1. Вступ до патернів проєктування

**Патерни проєктування (Design Patterns)** --- це типові, перевірені
рішення для поширених проблем у проєктуванні програмного забезпечення.\
Вони не є готовими бібліотеками чи фреймворками, а виступають
**шаблонами архітектурних рішень**, які можна адаптувати до конкретної
системи.

### Навіщо потрібні патерни

-   повторне використання перевірених підходів;
-   покращення комунікації між розробниками;
-   підвищення гнучкості та підтримуваності коду.

### Категорії патернів

-   **Породжувальні (Creational)** --- керують створенням об'єктів;
-   **Структурні (Structural)** --- описують композицію класів і
    об'єктів;
-   **Поведінкові (Behavioral)** --- визначають взаємодію між об'єктами.

У цій лекції розглядаються два породжувальні патерни: **Factory Method**
і **Singleton**.

------------------------------------------------------------------------

## 2. Factory Method (Фабричний метод)

### Призначення

Патерн **Factory Method** визначає інтерфейс для створення об'єкта, але
дозволяє підкласам вирішувати, **який саме клас інстанціювати**.

### Коли використовується

-   клас не знає наперед, які об'єкти потрібно створювати;
-   потрібно делегувати створення об'єктів підкласам;
-   необхідно зменшити залежність від конкретних класів.

### Структура

-   **Product** --- інтерфейс продукту;
-   **ConcreteProduct** --- конкретна реалізація;
-   **Creator** --- оголошує фабричний метод;
-   **ConcreteCreator** --- повертає конкретний продукт.

### Приклад (C#)

``` csharp
public interface ILogger { void Log(string message); }

public class ConsoleLogger : ILogger {
    public void Log(string message) {
        Console.WriteLine(message);
    }
}

public abstract class LoggerFactory {
    public abstract ILogger CreateLogger();
}
```

### Переваги

-   дотримання принципу **OCP**;
-   слабке зв'язування між клієнтом і конкретними класами;
-   розширюваність системи.

### Недоліки

-   збільшення кількості класів;
-   складніша структура коду.

### Реальні приклади використання

-   створення різних типів логерів;
-   підключення до різних баз даних;
-   генерація UI‑елементів під різні платформи.

------------------------------------------------------------------------

## 3. Singleton (Одинак)

### Призначення

Патерн **Singleton** гарантує існування **лише одного екземпляра класу**
та надає глобальну точку доступу до нього.

### Коли використовується

-   менеджер конфігурації;
-   логер застосунку;
-   пул підключень до бази даних;
-   кеш або глобальні налаштування.

### Типова структура

-   приватний статичний екземпляр;
-   приватний конструктор;
-   публічний статичний доступ до екземпляра.

### Переваги

-   контрольований доступ до єдиного об'єкта;
-   економія ресурсів;
-   централізоване керування станом.

### Недоліки

-   ускладнює модульне тестування;
-   може порушувати принцип **SRP**;
-   створює приховані глобальні залежності.

------------------------------------------------------------------------

## Висновок

-   **Factory Method** забезпечує **гнучке створення об'єктів** без
    залежності від їх конкретних класів.\
-   **Singleton** гарантує **єдиний екземпляр** важливого сервісного
    класу.

Обидва патерни широко застосовуються для побудови **масштабованих і
підтримуваних програмних систем**.
