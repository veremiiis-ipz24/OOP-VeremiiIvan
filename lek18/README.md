# Лекція №18

## Тема: Поведінкові патерни --- Strategy та Observer

------------------------------------------------------------------------

## 1. Патерн Strategy (Стратегія)

### Призначення

Патерн **Strategy** визначає сімейство алгоритмів, інкапсулює кожен з
них і робить їх взаємозамінними.\
Це дозволяє змінювати алгоритм незалежно від клієнтського коду, який
його використовує.

### Проблема

Виникає, коли: - існує кілька способів виконати одну і ту саму задачу; -
потрібно перемикати алгоритм під час виконання; - небажано
використовувати великі умовні конструкції (`if/else`, `switch`).

### Структура

-   **Strategy** --- інтерфейс спільного алгоритму.\
-   **ConcreteStrategy** --- конкретна реалізація алгоритму.\
-   **Context** --- клас, що використовує об'єкт Strategy.

### Приклад (C#)

``` csharp
public interface ISortStrategy { void Sort(List<int> data); }

public class BubbleSortStrategy : ISortStrategy { /* реалізація */ }

public class QuickSortStrategy : ISortStrategy { /* реалізація */ }

public class Sorter {
  private ISortStrategy _strategy;

  public Sorter(ISortStrategy strategy) { _strategy = strategy; }

  public void SetStrategy(ISortStrategy strategy) { _strategy = strategy; }

  public void PerformSort(List<int> data) { _strategy.Sort(data); }
}
```

### Переваги

-   Дотримання **OCP** --- нові алгоритми додаються без зміни існуючого
    коду.\
-   **Гнучкість** --- алгоритм можна змінювати під час роботи програми.\
-   **Інкапсуляція** --- логіка кожного алгоритму ізольована.

### Недоліки

-   Збільшується кількість класів.\
-   Клієнт повинен знати про існування різних стратегій.

### Реальні приклади використання

-   вибір способу оплати в інтернет‑магазині;
-   різні алгоритми сортування або стиснення даних;
-   поведінка персонажів у відеоіграх.

------------------------------------------------------------------------

## 2. Патерн Observer (Спостерігач)

### Призначення

Патерн **Observer** визначає залежність типу **один‑до‑багатьох** між
об'єктами.\
Коли стан **Subject** змінюється, всі підписані **Observer** автоматично
отримують сповіщення.

### Проблема

Потрібно: - повідомляти багато об'єктів про зміну стану; - уникати
жорсткої залежності між ними; - динамічно додавати або прибирати
підписників.

### Структура

-   **Subject** --- керує списком спостерігачів і надсилає
    повідомлення.\
-   **Observer** --- інтерфейс отримання оновлень.\
-   **ConcreteSubject** --- зберігає стан.\
-   **ConcreteObserver** --- реагує на зміни.

### Приклад (C#)

``` csharp
public interface IObserver { void Update(string message); }

public interface ISubject {
  void Attach(IObserver observer);
  void Detach(IObserver observer);
  void Notify(string message);
}

public class StockMonitor : ISubject { /* реалізація */ }

public class ConsoleDisplay : IObserver { /* реалізація */ }

public class EmailNotifierObserver : IObserver { /* реалізація */ }
```

### Переваги

-   **Слабке зв'язування** між компонентами.\
-   **Розширюваність** --- легко додати нові типи спостерігачів.\
-   **Динамічність** --- підписка/відписка під час виконання.

### Недоліки

-   Може виникати велика кількість повідомлень.\
-   Складніше відстежувати порядок оновлень.

### Реальні приклади використання

-   система підписок у соціальних мережах;
-   оновлення ціни акцій або криптовалют;
-   події графічного інтерфейсу (GUI).

------------------------------------------------------------------------

## Висновок

-   **Strategy** використовується для **заміни алгоритмів**.\
-   **Observer** --- для **реакції на події** та синхронізації стану
    об'єктів.

Обидва патерни підвищують **гнучкість, масштабованість і
підтримуваність** програмних систем.
